> **Trigger (paste to build):**
> `/loop /dev-loop CLIENT_ONBOARDING_IMPORT_HANDOFF.md`
>
> Plan gate: batched · Merge mode: per-phase

> **v2 supersedes v1.** v1 scoped a fixed 4-entity-type tool (venue/pitches/teams/staff/players), one bespoke RPC per type. After review, the operator widened this to a generic, extensible import ENGINE with a growing CATALOG of importable data kinds, a swappable access-control gate (designed for, not built, a future self-serve flip), save-and-resume, a completeness/coverage nudge, and sport/club-type qualifying questions. ---

## WHAT IT IS

Today every new venue/club is created by hand: Tarny runs an onboarding call, then re-types the client's pitches, teams, staff and rosters one screen at a time into `/superadmin/venues/new`. Real clients almost always already have this data — a messy Pitchero/TeamApp export or a spreadsheet — and increasingly it's not just rosters: equipment lists, membership records, class timetables, existing bookings, past-season results.

This tool lets Tarny paste or upload that data during the call and get it auto-provisioned, with a plain-English preview before anything commits. The key shift from v1: instead of 4 hardcoded boxes, the tool is built as a **generic engine** (upload → classify → map → validate → preview → commit) driven by a **catalog** of importable data types — so adding "equipment" or "memberships" to what the tool understands is a data addition to the catalog, not a rewrite of the wizard or the RPC. The engine also doesn't know or check *who* is calling it — a separate, thin **gate** layer resolves that — so a future decision to let a trusted venue/club admin self-import their own data into their own tenant is a contained change to the gate, not a rewrite of the engine. **That gate stays platform-admin-only in this build — no self-serve is being opened up now**, this is architecture, not a feature flag flip.

**Two new UX ideas folded in from operator discussion:**
- **A qualifying step 0** — before any data is touched, ask what kind of client this is (single football team / multi-team club / gym / boxing club / another sport) and whether under-18s are involved. This isn't cosmetic: it filters which catalog entries the wizard even shows (a boxing gym never sees "league fixtures"; a five-a-side team never sees "belts/grading"), drives field-name labels ("squad" vs "class" vs "session"), and flags the compliance path (minors) upfront instead of discovering it mid-import. The platform already supports this — `venues.sport`/`playing_areas.sport` and the shipped Gym/boxing vertical (sparring, grading/belts, PT, fight record — migs 355–359) are exactly this distinction, just never asked as an onboarding question before.
- **A completeness nudge, not a checklist** — near the end, one plain-English line: "You've told us about teams and players — most clubs like this also have equipment and membership types, want to add those now?" Sourced from the platform's own real feature catalog (`apps/venue/src/views/FeaturesView.jsx`), never a rendered 9-item form (that reads as a compliance exercise, not a helpful nudge, for a non-technical operator on a live call).

## LOCKED DECISIONS

Confirm or adjust at review. v1's decisions are folded in and renumbered where restated more generally; **(new)** marks what's added this pass.

1. **Stays platform-admin-only.** No new access surface opens in this build. (Unchanged from v1 — DECISIONS.md session 48 still fully in force.)
2. **Server-side dry-run, run PER ENTITY TYPE immediately after that type's mapping is confirmed — not deferred to one pass at the end.** Client-only validation only catches basic shape errors; the real check (dedup collisions, FK dependencies) needs the DB. Critically, each type's dry-run checks against BOTH the live database AND any other entity types already staged earlier in the same draft (e.g. validating players against teams just confirmed in the previous step, even though those teams aren't written to the database yet). This means the operator sees and fixes problems with each data type as soon as it's entered, not after pasting everything — the exact opposite of discovering a teams-sheet formatting problem only after also entering staff and 140 players. One final, fast confirmation dry-run still runs immediately before the actual commit as a safety net (catches anything that changed since the per-type checks, e.g. a clashing email created by someone else in the meantime) — but it's a formality by that point, not the first time anything gets checked.
3. **Row-level partial failure inside one transaction** — one bad row never blocks the rest.
4. **No fuzzy auto-merge of people** — exact email match only; everything else previewed as "new" with an operator-set skip/merge override.
5. **Staff import as pending-invite, never auto-active** — real login always via the person's own sign-in.
6. **Imported people reuse the existing unclaimed-profile pattern** (`auth_user_id`/`user_id` NULL) — **(generalized this pass, see #13)**.
7. **Rule-based fuzzy header matching for v1**, manual dropdown override always available; true AI-assist is later, only if needed.
8. **Validation errors are always pre-translated plain English**, never raw SQL.
9. **Player/member import ships as its own gated PR**, separate from the entity-agnostic engine.
10. **Re-import uses upsert-by-natural-key** — re-uploading a corrected file updates existing rows, never duplicates.
11. **Every commit is tagged with an `import_batch_id`**, referenced by every row it writes — the reversal and audit mechanism.
12. **Legal basis for bulk-imported minors' data sits with the venue/club (contract/legitimate interest), never the import act itself** — mirrors UK ICO guidance for clubs and the FERPA "school official exception" precedent; a guardian-facing consent event stays separate and later.
13. **(new) The "represents a real person" rule is a property of the TARGET TABLE, enforced generically, not hand-written per entity type.** Any catalog entry whose target table has an identity/consent column (`auth_user_id`, `user_id`, a `consent_*` field) is flagged `represents_person: true` in the registry; the generic commit function hard-blocks any payload attempt to set those columns, for every entity type that carries the flag — players, staff, and (when built) membership-holders all inherit this automatically, rather than each needing its own hand-written safeguard.
14. **(new) Qualifying step 0 (sport/club-type + under-18 flag) runs before any data is uploaded.** Answers filter the catalog shown and set field-name labels; the under-18 flag is recorded on the `import_batch_id` and feeds the heavier compliance gate on PR4 regardless of which entity types are actually imported.
15. **(new) "What are you importing?" is open paste-and-auto-classify, not a fixed checklist.** A non-technical operator on a live call finds "paste everything, confirm our guess" far lower cognitive load than pre-sorting a client's spreadsheet into named boxes before pasting anything. The qualifying step (14) already narrows the guess space. Auto-classify reuses the same rule-based-guess-plus-override mechanism as column mapping (#7), just applied one level up.
16. **(new) Progressive disclosure — the wizard only ever shows steps for entity types actually selected/detected.** The catalog's breadth is a menu, not a mandatory tour; someone who only has a roster sees exactly the same shape as the original 4-box design. This is the direct mitigation for "9 catalog types = overwhelming."
17. **(new) The completeness nudge is one skippable, plain-English line near the end, sourced from `apps/venue/src/views/FeaturesView.jsx`'s real `VENUE_FEATURES`/`CLUB_FEATURES` catalog** (bookings/spaces/room_hire/equipment/memberships/competition/club_leagues/coaching/tournaments/public_web) — never a rendered checklist. Never blocking.
18. **(new) Gate/engine separation is structural, not cosmetic.** A single `_resolve_import_actor_scope()` function (called once, at the top of the commit RPC) is the ONLY place "who is allowed to do this, for which tenant" is decided. The rest of the engine — catalog lookup, dependency ordering, per-row upsert — operates purely on the scope record it returns and never re-checks identity. **The future self-serve variant is a brand-new sibling RPC with its own `_resolve_import_actor_scope()` body and its own gated EV/security-sweep cycle — never a parameter flip on this RPC, and never a runtime-switchable config.** Any future change of an RPC's gate FROM platform-admin TO a tenant-scoped check is automatically tier-3/PROTECTED, no exceptions, even if the RPC body is otherwise unchanged — write this as a standing rule, not a one-off judgment call.
19. **(new) The catalog registry is data-driven for field/synonym/dedup metadata, but the allowlist of writable target tables and their actual INSERT statements stays hardcoded, literal SQL inside the commit RPC, changed only by a reviewed migration.** Never build a fully dynamic `EXECUTE format('INSERT INTO %I ...', target_table)` engine — a registry row is data an operator could theoretically influence; if it could also decide which table gets written, a mis-seeded or compromised row could point at a sensitive table and defeat RLS entirely. Precedent for the safe middle ground already exists in this codebase: `venue_list_spaces` (RPCS.md ~L835) uses `to_regclass`-guarded dynamic SQL that self-upgrades as new tables are added, while the *set* of tables it can ever touch stays enumerated and migration-reviewed.
20. **(new) Return shape convention, locked now to prevent a mig-070-class latent bug:** `{ok, summary: {by_entity_type: {...}}, rows: [{index, entity_type, status, reason}], warnings: []}`. Adding a new catalog entry only adds a new key under `summary.by_entity_type` and new `entity_type` values in `rows` — never restructures an existing key. The `packages/core/storage/supabase.js` mapper for this RPC's return value must be updated in the SAME commit as any new field (Hard Rule #12).
21. **(new) Draft/resume storage gets the same treatment as committed data, plus a hard, short retention window.** `import_drafts` rows (raw uploaded data + mapping state, potentially including children's names/DOB) auto-expire and are hard-deleted after **7 days**, not soft-flagged, with a visible countdown in the UI. Same `is_platform_admin()`-only access as everything else. Reasoning: a draft has no lawful basis attached yet (unlike a committed row, which inherits the club's contract/legitimate-interest basis at commit time) — indefinite "just in case" storage of unconsented PII is a bigger liability than asking the operator to re-upload if a pause runs long.
22. **(new) Membership import (when built) imports CURRENT state only, never fabricates historical billing/payment history.** There's no schema destination for historical charges, and a fabricated `renews_at`/active-membership row would be picked up by the real billing cron (`run_membership_renewals`) — safe only if the imported row reflects genuine ongoing status, never invented.
23. **(new) Historical results/fixtures import is explicitly a future, gated decision — not scoped as engineering work in this build.** `fixtures.competition_id` is a required FK into a `competitions`→`seasons`→`leagues` chain normally created fresh for this platform, not carried over. Before this is buildable at all, the operator must decide: does the tool auto-create a placeholder "Imported History" competition shell, or does the operator create one via the existing UI first? This is a product call, not something to default silently.

24. **(new, from flow gap-hunt) Going back to edit an earlier entity type invalidates every later type's validation that depended on it.** If the operator returns to "teams" after "players" already passed dry-run validation, the players validation result is marked stale and MUST re-run before commit is allowed — Decision 2's "validate against types staged earlier" cuts both ways: it also means a change to an earlier type can silently break a later type's already-passed check if this isn't enforced.
25. **(new) The wizard auto-sequences detected/pasted entity types by dependency order, regardless of the order the operator pasted or classified them in.** Decision 15's open paste-and-classify never asked the operator to paste in a particular order — so the wizard itself (using the same dependency chain PR1's registry encodes, e.g. teams before players) decides validation/commit order, not the paste order. Pasting players before teams must not silently attempt to validate players with no teams to check against.
26. **(new) The commit RPC is one single all-or-nothing database transaction at the SQL level.** A client-side failure mid-call (network drop, timeout) is always safe to retry using the same `import_batch_id` — the upsert-by-natural-key behavior (Decision 10) means a retry after an unknown-outcome failure re-converges to the correct state rather than double-importing. This is a clarification of Decision 3, not a new mechanism.
27. **(new) Changing a Step 0 qualifying answer (Decision 14) after any data is already staged forces a confirm-and-reset of everything beyond Step 0.** The catalog subset and field labels later steps depend on change underneath them the moment sport/club-type changes — silently keeping stale data staged against a since-changed catalog is not safe. The operator sees an explicit "this will clear what you've entered so far" confirmation, not a silent wipe.
28. **(new) Resuming a paused draft (PR3) always re-runs per-type dry-run validation before commit is allowed, never trusts the stored pass/fail state as-is.** A draft paused for hours or days can go stale against the live database exactly along the dimensions Decision 2's checks care about (dedup collisions, FK targets) — a team renamed or an email claimed elsewhere while paused. Cheap to re-check, expensive to silently import against stale assumptions.
29. **(new) The completeness/coverage nudge (Decision 17) only ever offers catalog entries that actually exist in the registry at that point in the rollout, filtered against `import_catalog_entries` — never the full `FeaturesView.jsx` feature list.** PR2 ships before PR5 (equipment) and PR6+ (memberships/bookings/history); if the nudge suggested "add equipment now" before PR5 exists, the operator would hit a dead end with nowhere for that data to go. The nudge's copy source is always the live, currently-buildable subset, not the platform's full aspirational feature list.
30. **(new) Editing a single row's mapping or data on the issues screen after that entity type's dry-run has already passed clears that type's "validated" flag and requires re-running its dry-run** — it does not silently carry through to commit on the strength of the earlier pass, and it must not rely solely on the final pre-commit safety-net check (Decision 2 explicitly calls that check "a formality by that point," so it cannot be the only thing catching a late edit).

**Open questions for the operator (defaults stand if unanswered), including two surfaced by the SWEEP pass below:**
- **(new) Export/reversal for a client leaving the platform, or a GDPR data-portability request.** `import_batch_id` (Decision #11) is currently framed only as an audit/upsert-dedup mechanism, never as "delete everything this batch created" or "hand the client their data back." Decision #12 grounds the legal basis in the club's own contract, which implies an offboarding duty this doc doesn't yet answer. *(Not scoped as a PR here — flagged so it isn't silently assumed solved. `venue_erase_customer`'s erase-but-retain-row pattern, already cited elsewhere in this doc, is the likely shape when this gets built.)*
- **(new) Non-tabular source data.** The whole engine assumes delimited/tabular paste (Decision #15). Real onboarding calls sometimes produce a PDF, a WhatsApp export, or a photo of a whiteboard fixture list — none of which fit "paste and auto-classify" as designed. *(Not a v1 blocker — flagged as a known limitation of the paste-based design, worth revisiting if it turns out to be common rather than rare.)*
- Payment-history import: a club switching platforms after a dispute might have people who should NOT be re-contacted — should there be a per-row "import but suppress all comms" flag, separate from the skip/merge dedup override? *(Flagged, not resolved — no existing suppression-list precedent in this codebase to reuse.)*
- Should the qualifying step's sport/club-type answer be a fixed list (football/multi-team club/gym/boxing/other) or open text the tool classifies? *(Default: fixed list — mirrors the "checklist beats free classification" reasoning used for the coverage step, but here the classification decision affects field labels the operator will read throughout the call, so getting it visibly right matters more than in the completeness nudge.)*
- Do class/room-hire attendee imports (which key on `member_profiles`, a structurally separate identity table from `venue_customers`) get their own future import track, or should the platform bridge `venue_customers`↔`member_profiles` (`source_customer_id`) as part of that later PR? *(Flagged as a genuine data-model open question, not a v1 blocker — no entity type touching `member_profiles` is in this build's roadmap.)*

## MISSED / OPPORTUNITY / FUTURE-PROOF / WOW

**Missed:** No export/reversal path for a client leaving the platform or a GDPR data-portability request — `import_batch_id` only does audit/dedup today, not "give this data back" (now an open question above). No stated ownership/handoff model for a paused draft — `import_drafts` is `is_platform_admin()`-gated but not per-admin-scoped, so it's implicitly "any platform admin can see/resume any draft," never stated as a deliberate choice (worth confirming that's actually what's wanted, e.g. if a colleague ever gets platform-admin access). No consideration of non-tabular real-world sources (PDF, WhatsApp export, a photo of a whiteboard) — the whole engine assumes delimited/tabular paste, which is the common case but not the only one (now an open question above).

**Opportunity:** This isn't a stretch — STRATEGY.md's deferred **Phase C "FA AI-import"** (read messy external league data → classify/map → one-time mapping → upsert into `club_fixtures` with a `source`/diff key, gated on a real FA snippet) is structurally the same shape as this engine: classify → map → dry-run/validate → upsert-by-natural-key → provenance-tagged commit. The direction differs (Phase C polls a live external source on a schedule; this tool ingests a human-supplied file once), but the registry pattern (Locked Decision 19) and the upsert/provenance machinery (Decisions 10-11) could plausibly become the substrate Phase C's ingest calls into once a `fixtures` catalog entry exists — rather than building two separate systems later. Worth a note to whoever eventually scopes Phase C: check this engine first.

**Future-proofed option:** Locked Decision 18 — the `_resolve_import_actor_scope()` gate/engine split — buys the most future flexibility for the least cost paid now (one extra function-call indirection instead of an inline check). It's what makes the eventual self-serve venue/club-admin variant a brand-new sibling RPC with its own scope resolver rather than a parameter flip, and it comes with a built-in guardrail against the way this kind of abstraction usually gets quietly loosened under deadline pressure (any future platform-admin→tenant-scoped gate change is automatically tier-3/PROTECTED, no exceptions).

**Wow factor(s):** Genuinely absent until this pass — the whole document is optimized for Tarny's side of the call. The one addition now folded into PR2 (a client-facing plain-English completion summary, "we've set up 3 teams, 47 players and 2 pitches from your file") is the closest thing to a wow moment for the person on the other end of the phone, and it's nearly free since it reuses the RPC's existing return shape rather than needing new data plumbing. If there's a real second wow-factor beyond that, it hasn't surfaced yet — worth revisiting once the tool's actually used a few times and it's clear what clients react to.
